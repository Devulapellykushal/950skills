# Idempotency for APIs

## When to use

Use this skill when the user needs **idempotent API design** for payments, mutations, or retries. Triggers: "idempotency", "idempotency key", "duplicate request", "retry safe", "payment API", "Idempotency-Key".

## Overview

Implement idempotency for critical APIs so that duplicate requests (retries, replays) do not cause duplicate side effects; use idempotency keys and server-side state.

## Deliverables

1. **Idempotency key** – Client sends a unique key (e.g. Idempotency-Key header); server uses it to recognize repeat requests; key scope (per client, per resource, or global).
2. **Storage** – Store key → response (and optionally request hash) in cache or DB; TTL (e.g. 24h) to avoid unbounded growth; partition by client or key prefix.
3. **Behavior** – First request: execute, store result, return. Subsequent same key: return stored result without re-executing; same key with different body: reject (409 or 422).
4. **Scope** – Define what the key applies to: whole request body, or method + resource; document in API spec.
5. **Cleanup** – TTL expiry; optional admin endpoint to purge; consider persistence for financial or compliance needs.

## Principles

- Keys must be generated by the client and unique per logical operation.
- Never execute a mutation twice for the same idempotency key.
- Document key format, TTL, and error responses in API docs.
